# Документация страницы: Заявки на подбор (hiring-requests/page.tsx)

## Общее описание

Страница управления заявками на подбор персонала. Позволяет создавать, просматривать, фильтровать и группировать заявки в двух режимах отображения: таблица ("Все") и блоки ("Блоками"). Включает статистику по заявкам, поиск, фильтрацию и импорт/экспорт в Excel.

**Путь**: `/hiring-requests`

## Компоненты страницы

### 1. HiringRequestsPage (основной компонент)
- **Расположение**: `@/app/hiring-requests/page.tsx`
- **Назначение**: Главный компонент страницы управления заявками на подбор
- **Функциональность**: 
  - Управление состоянием фильтров и режима отображения
  - Фильтрация заявок по поисковому запросу, рекрутеру, статусу и приоритету
  - Группировка заявок по названию вакансии в режиме "Блоками"
  - Вычисление статистики по заявкам
  - Управление модальным окном создания заявки
- **Связи**: 
  - RequestsSearchFilters: компонент поиска и фильтров
  - RequestsStats: компонент статистики
  - RequestsTable: таблица заявок (режим "Все")
  - RequestListItem: элемент списка заявок (режим "Блоками")
  - CreateRequestModal: модальное окно создания заявки

## Интерфейсы и типы

### Request (основной интерфейс заявки)
```typescript
interface Request {
  id: number                                    // Уникальный идентификатор заявки
  title: string                                 // Название вакансии
  grade: string                                 // Грейд (Junior+, Middle, Middle+ и т.д.)
  project: string | null                         // Проект (может быть null)
  recruiter: string                             // Рекрутер, ведущий заявку
  recruiterDays: number                         // Количество дней работы рекрутера над заявкой
  status: 'planned' | 'in_process' | 'closed' | 'cancelled'  // Статус заявки
  statusDate?: string                           // Дата изменения статуса (для закрытых/отмененных)
  startDate: string                             // Дата начала подбора (ДД.ММ.ГГГГ)
  endDate: string                               // Дата окончания подбора (ДД.ММ.ГГГГ)
  isOverdue?: boolean                           // Флаг просрочки заявки
  factDays: number                              // Фактическое количество дней подбора
  slaDays: number                               // SLA по количеству дней
  slaStatus: 'normal' | 'risk' | 'overdue' | 'on_time'  // Статус SLA
  t2hDays?: number                              // Время до найма в днях (опционально)
  t2hSlaDays?: number                           // SLA по времени до найма
  candidate?: {                                 // Информация о найденном кандидате (опционально)
    name: string
    id: string
  }
  department: string                            // Отдел/департамент
  priority: 'high' | 'medium' | 'low'          // Приоритет заявки
  technologies: string[]                       // Массив технологий
  candidates: number                            // Количество кандидатов
  date: string                                  // Дата создания заявки (ДД.ММ.ГГГГ)
  hasWarning: boolean                           // Флаг наличия предупреждения
  warningText?: string                          // Текст предупреждения (опционально)
}
```

**Статусы заявок**:
- `planned`: Планируется (синий Badge)
- `in_process`: В процессе (оранжевый Badge)
- `cancelled`: Отменена (красный Badge)
- `closed`: Закрыта (зеленый Badge)

**Статусы SLA**:
- `normal`: Нормально (серый Badge)
- `risk`: Риск просрочки (оранжевый Badge)
- `overdue`: Просрочено (красный Badge)
- `on_time`: В срок (зеленый Badge)

**Приоритеты**:
- `high`: Высокий (красный Badge)
- `medium`: Средний (желтый Badge)
- `low`: Низкий (серый Badge)

## Состояние компонента

### State переменные:
- `displayMode` ('all' | 'blocks'): режим отображения
  - Инициализация: `'blocks'` (режим "Блоками" по умолчанию)
  - Обновление: при переключении режима через SegmentedControl
  - Использование: определение, какой компонент использовать для отображения
- `searchQuery` (string): поисковый запрос для фильтрации заявок
  - Инициализация: `''`
  - Обновление: при вводе в поле поиска
  - Использование: фильтрация заявок по названию, ID или отделу
- `selectedRecruiter` (string): выбранный рекрутер для фильтрации
  - Инициализация: `'all'` (все рекрутеры)
  - Обновление: при выборе рекрутера в Select
  - Использование: фильтрация заявок по рекрутеру
- `selectedStatus` (string): выбранный статус для фильтрации
  - Инициализация: `'all'` (все статусы)
  - Обновление: при выборе статуса в Select
  - Использование: фильтрация заявок по статусу
- `selectedPriority` (string): выбранный приоритет для фильтрации
  - Инициализация: `'all'` (все приоритеты)
  - Обновление: при выборе приоритета в Select
  - Использование: фильтрация заявок по приоритету
- `isCreateModalOpen` (boolean): флаг открытия модального окна создания заявки
  - Инициализация: `false`
  - Обновление: при открытии/закрытии модального окна
  - Использование: управление видимостью CreateRequestModal

### Hooks:
- `useRouter()`: программная навигация
  - Используется для: перехода на детальную страницу заявки или редактирование

## Функции и обработчики

### Вычисление статистики
- `totalRequests`: общее количество заявок (`mockRequests.length`)
- `plannedRequests`: количество заявок со статусом 'planned'
- `inProcessRequests`: количество заявок со статусом 'in_process'
- `cancelledRequests`: количество заявок со статусом 'cancelled'
- `closedRequests`: количество заявок со статусом 'closed'

### filteredRequests
- **Назначение**: Отфильтрованный список заявок
- **Логика**: 
  1. Поиск: проверяет наличие запроса в названии, ID или отделе (без учета регистра)
  2. Фильтр по рекрутеру: если выбран 'all' - пропускает все, иначе проверяет точное совпадение
  3. Фильтр по статусу: если выбран 'all' - пропускает все, иначе проверяет совпадение статуса
  4. Фильтр по приоритету: если выбран 'all' - пропускает все, иначе проверяет совпадение приоритета
  5. Все четыре условия должны выполняться одновременно (логическое И)
- **Поведение**: 
  - Вычисляется при каждом рендере компонента
  - Используется для отображения в таблице или блоках
- **Связи**: 
  - Используется в RequestsTable и RequestListItem

### groupedRequests
- **Назначение**: Объект сгруппированных заявок по названию вакансии
- **Логика**: 
  1. Используется только в режиме `displayMode === 'blocks'`
  2. Создает объект, где ключ - название вакансии (title), значение - массив заявок
  3. Использует reduce для группировки
- **Структура**: `Record<string, Request[]>` (ключ - название вакансии, значение - массив заявок)
- **Поведение**: 
  - Позволяет видеть все заявки на одну и ту же вакансию вместе
  - Используется для отображения в режиме "Блоками"
- **Связи**: 
  - Используется в RequestListItem для отображения групп заявок

## Режимы отображения

### Режим "Все" (displayMode === 'all')
- **Компонент**: RequestsTable
- **Особенности**: 
  - Отображает все заявки в табличном виде
  - Полная информация в колонках
  - Сортировка по ID (от новых к старым)
  - Подходит для детального анализа всех заявок
- **Колонки таблицы**:
  - Вакансия / Грейд: название вакансии и грейд в виде тега
  - Проект: название проекта или "—"
  - Рекрутер: имя рекрутера и количество дней работы
  - Статус: Badge статуса с цветовой индикацией и датой изменения (если не in_process)
  - Сроки: даты начала и окончания подбора
  - Факт/SLA: фактические дни / SLA дни и Badge статуса SLA
  - T2H | SLA: время до найма / SLA по времени до найма
  - Кандидат: имя и ID найденного кандидата или "—"
  - Действия: кнопки просмотра и редактирования

### Режим "Блоками" (displayMode === 'blocks')
- **Компонент**: RequestListItem
- **Особенности**: 
  - Группирует заявки по названию вакансии
  - Показывает основную информацию о группе
  - Разворачиваемая таблица с деталями всех заявок в группе
  - Развернута по умолчанию для групп заявок
  - Подходит для работы с несколькими заявками на одну вакансию
- **Структура элемента списка**:
  - Заголовок: название вакансии, Badge статуса, Badge приоритета, ID заявки
  - Информация: отдел, рекрутер, количество заявок в группе, количество кандидатов, дата, предупреждения
  - Кнопки действий: разворот (если есть данные), просмотр, редактирование
  - Развернутая таблица: детальная информация о всех заявках в группе (если развернута)

### Переключатель режимов
- **Компонент**: SegmentedControl из Radix UI
- **Опции**: "Все" (value="all"), "Блоками" (value="blocks")
- **Логика**: При изменении перерисовывает список заявок
- **Поведение**: 
  - При переключении на "Все" отображает RequestsTable
  - При переключении на "Блоками" группирует заявки и отображает RequestListItem

## Фильтрация и поиск

### Поиск (searchQuery)
- **Поле**: TextField в RequestsSearchFilters
- **Placeholder**: "Поиск по названию, ID или отделу..."
- **Логика**: 
  - Ищет по названию заявки (без учета регистра)
  - Ищет по ID заявки (точное совпадение в строке)
  - Ищет по отделу (без учета регистра)
- **Поведение**: 
  - Фильтрация происходит в реальном времени при вводе
  - Обновляет filteredRequests при каждом изменении
- **Использование**: 
  - Используется в вычислении filteredRequests

### Фильтр по рекрутеру (selectedRecruiter)
- **Поле**: Select в RequestsSearchFilters
- **Опции**: "Все рекрутеры" (value="all"), список рекрутеров
- **Логика**: 
  - Если 'all' - пропускает все заявки
  - Иначе проверяет точное совпадение имени рекрутера
- **Поведение**: 
  - Обновляет filteredRequests при изменении
- **TODO**: Загружать список рекрутеров из API

### Фильтр по статусу (selectedStatus)
- **Поле**: Select в RequestsSearchFilters
- **Опции**: 
  - "Все статусы" (value="all")
  - "Планируется" (value="planned")
  - "В процессе" (value="in_process")
  - "Отменена" (value="cancelled")
  - "Закрыта" (value="closed")
- **Логика**: 
  - Если 'all' - пропускает все заявки
  - Иначе проверяет совпадение статуса заявки
- **Поведение**:

# API Specifications Index

## Общее описание

Эта директория содержит детальные спецификации для различных аспектов API HR Helper. Документы связаны перекрёстными ссылками и приведены к единому стилю без противоречий; при расхождении приоритет у [BUSINESS_LOGIC.md](./BUSINESS_LOGIC.md). Сводка по всем спецификациям: **[SUMMARY.md](./SUMMARY.md)**.

## Структура документации

### 1. Business Logic
**Файл:** [`BUSINESS_LOGIC.md`](./BUSINESS_LOGIC.md)

**Содержание:**
- Workflow переходы между этапами, валидация, исключения, бизнес-правила
- Алгоритмы (matching кандидатов, назначение интервьюеров), события и уведомления
- **Бизнес-логика 54+ страниц и модулей:** Workflow, Чат рекрутера, AI Чат, Календарь, Вики, Вакансии, Заявки, настройки компании, Финансы, Отчетность, Telegram, Huntflow, Поиск, детальные страницы
- **Страницы аккаунта:** Вход (email/пароль, Google OAuth), восстановление и сброс пароля, Профиль (вкладки: просмотр, редактирование, интеграции, быстрые кнопки; API профиля/интеграций/темы)
- **Страницы ошибок:** 401, 402, 403, 404, 500
- **Переиспользуемые компоненты и части страниц:** Layout, общие UI, плавающие действия, паттерны страниц, компоненты по доменам
- Дополнительные процессы: резюме, Scorecard, офферы, зарплатные вилки

**Статус:** ✅ Полностью проработано (версия 2.0.0)

## Как использовать эти спецификации

### Для Backend разработчиков

1. **Изучите все документы** перед началом разработки
2. **Следуйте стандартам:**
   - Формат ошибок из `ERROR_HANDLING.md`
   - Структура БД из `DATABASE_SCHEMA.md`
   - Правила доступа из `AUTHENTICATION_AUTHORIZATION.md`
3. **Реализуйте бизнес-логику** согласно `BUSINESS_LOGIC.md`
4. **Настройте интеграции** по `INTEGRATIONS.md`
5. **Оптимизируйте производительность** согласно `PERFORMANCE_SCALABILITY.md`

### Для Frontend разработчиков

1. **Используйте спецификации** для понимания API
2. **Реализуйте обработку ошибок** по `ERROR_HANDLING.md`
3. **Следуйте правилам аутентификации** из `AUTHENTICATION_AUTHORIZATION.md`
4. **Учитывайте ограничения** из `PERFORMANCE_SCALABILITY.md` (rate limiting, pagination)

### Для Product/QA

1. **Используйте бизнес-логику** из `BUSINESS_LOGIC.md` для тестирования
2. **Проверяйте права доступа** согласно матрице в `AUTHENTICATION_AUTHORIZATION.md`
3. **Тестируйте обработку ошибок** по примерам из `ERROR_HANDLING.md`

Все документы используют семантическое версионирование:
- **Major** (1.0.0) - критические изменения, ломающие обратную совместимость

## Обновления (версия 1.1.0)

### Новые функции:

1. **Этапы найма с настройками встреч:**
   - Добавлены поля `is_meeting`, `show_offices`, `show_interviewers` в таблицу `recruiting_stages`
   - Этапы с `is_meeting = true` используются для формирования динамических тогглеров на страницах `/workflow` и `/recr-chat`
   - Панель настроек встречи отображается условно в зависимости от настроек этапа
   - Новые права доступа: `read:recruiting_stages`, `write:recruiting_stages`

2. **Динамические тогглеры этапов процесса:**
   - Тогглеры формируются динамически из этапов найма с меткой "встреча"
   - Кнопка "Скрининг" всегда доступна (30 минут)
   - Количество кнопок: 0 и более (зависит от количества этапов с `is_meeting = true`)
   - Названия кнопок берутся из названий этапов найма

3. **Команды рекрутинга:**
   - Добавлена таблица `recruiting_commands`
   - Команда должна быть обязательно связана с этапом найма (`stage_id` обязателен)
   - Поддержка работы в любой раскладке клавиатуры (`allow_any_layout`)
   - Типы команд: `analysis` (анализ) или `event` (событие)
   - Проверка уникальности команд с учетом раскладки клавиатуры
   - Новые права доступа: `read:recruiting_commands`, `write:recruiting_commands`, `delete:recruiting_commands`
   - Новые коды ошибок: `COMMAND_DUPLICATE`, `COMMAND_INVALID_FORMAT`, `COMMAND_STAGE_REQUIRED`, `COMMAND_TYPE_INVALID`, `COMMAND_STAGE_NOT_FOUND`

4. **Настройки вакансии по странам:**
   - Добавлены таблицы `vacancy_text_by_country`, `vacancy_activity_by_country`, `vacancy_questions_by_office`
   - Текст вакансии настраивается отдельно для каждой страны
   - Активность вакансии управляется отдельно для каждой страны
   - Вопросы и ссылки настраиваются отдельно для каждого офиса
   - Автоматическая синхронизация активности между странами и офисами
   - Новые права доступа: `read:vacancy_country_settings`, `write:vacancy_country_settings`
   - Новые коды ошибок: `VACANCY_COUNTRY_NOT_FOUND`, `VACANCY_COUNTRY_INACTIVE`, `VACANCY_FIELD_INACTIVE`, `VACANCY_OFFICE_NOT_FOUND`, `VACANCY_OFFICE_INACTIVE`

5. **Производительность:**
   - Кэширование этапов-встреч (TTL: 10 минут)
   - Кэширование настроек вакансии по странам (TTL: 5 минут)
   - Оптимизированные запросы с JOIN для загрузки этапов с настройками
   - Индексы на `is_meeting`, `country_code`, `office_id` для быстрого поиска

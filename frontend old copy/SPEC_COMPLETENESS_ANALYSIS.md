# Анализ полноты покрытия спецификаций API

## Общее описание

Этот документ содержит результаты анализа полноты покрытия моковых данных API эндпоинтами для всех спецификаций проекта.

## Методология анализа

Анализ выполняется по следующим критериям:

1. **Извлечение сущностей из моковых данных**:
   - Поиск названий в формате `mockEntityName`
   - Извлечение типов из описаний (`Array<Type>`, `массив Type`)
   - Анализ интерфейсов TypeScript

2. **Анализ API эндпоинтов**:
   - Группировка эндпоинтов по сущностям (из путей `/api/entity/`)
   - Проверка наличия CRUD операций (GET, POST, PUT/PATCH, DELETE)

3. **Оценка покрытия**:
   - Для каждой сущности проверяется наличие всех необходимых операций
   - Вычисляется процент полностью покрытых сущностей

## Запуск анализа

```bash
cd frontend
python3 analyze_spec_completeness.py
```

Скрипт создаст файл `detailed_coverage_report.json` с детальными результатами.

## Интерпретация результатов

### Покрытие 100%
- Все моковые данные имеют соответствующие API эндпоинты
- Все необходимые CRUD операции присутствуют

### Покрытие 50-99%
- Большинство сущностей покрыты, но есть недостающие операции
- Рекомендуется добавить недостающие эндпоинты

### Покрытие <50%
- Значительная часть моковых данных не имеет соответствующих эндпоинтов
- Требуется детальный анализ и добавление недостающих эндпоинтов

## Рекомендации по улучшению

### Для спецификаций с низким покрытием:

1. **Проверьте соответствие названий**:
   - Убедитесь, что названия сущностей в моковых данных совпадают с путями эндпоинтов
   - Например: `mockVacancies` → `/api/vacancies/`

2. **Добавьте недостающие CRUD операции**:
   - GET - для получения списка/одного элемента
   - POST - для создания
   - PUT/PATCH - для обновления
   - DELETE - для удаления (если применимо)

3. **Проверьте описание эндпоинтов**:
   - Убедитесь, что метод и путь указаны в правильном формате
   - Формат: `GET /api/entity/` или `POST /api/entity/`

4. **Обновите спецификации**:
   - Добавьте недостающие эндпоинты в раздел `api_endpoints`
   - Используйте предложения из отчета как основу

## Примеры исправлений

### Пример 1: Добавление недостающего GET эндпоинта

**Было:**
```json
{
  "mock_data": {
    "mockVacancies": {
      "description": "Массив вакансий"
    }
  },
  "api_endpoints": [
    {
      "method": "POST",
      "endpoint": "/api/vacancies/",
      "description": "Создание вакансии"
    }
  ]
}
```

**Стало:**
```json
{
  "mock_data": {
    "mockVacancies": {
      "description": "Массив вакансий"
    }
  },
  "api_endpoints": [
    {
      "method": "GET",
      "endpoint": "/api/vacancies/",
      "description": "Получение списка вакансий",
      "parameters": "page, page_size, search (опционально)",
      "returns": "Массив объектов Vacancy"
    },
    {
      "method": "POST",
      "endpoint": "/api/vacancies/",
      "description": "Создание вакансии"
    }
  ]
}
```

## Следующие шаги

1. ✅ Запустить анализ всех спецификаций
2. 🔄 Просмотреть отчет `detailed_coverage_report.json`
3. 🔄 Для каждой спецификации с низким покрытием:
   - Изучить недостающие операции
   - Добавить недостающие эндпоинты в спецификацию
   - Обновить документацию при необходимости
4. 🔄 Повторно запустить анализ для проверки улучшений

## Автоматическое обновление спецификаций

После анализа можно использовать предложения из отчета для автоматического добавления недостающих эндпоинтов:

```python
# Пример использования предложений из отчета
import json

with open('detailed_coverage_report.json', 'r') as f:
    report = json.load(f)

for result in report['results']:
    if result['coverage_score'] < 100:
        # Добавить suggestions в спецификацию
        spec_file = result['file']
        # ... обновить спецификацию
```

## Примечания

- Некоторые сущности могут не требовать всех CRUD операций (например, настройки, конфигурации)
- Некоторые эндпоинты могут быть описаны в тексте, но не в структурированном формате
- Анализ основан на автоматическом сопоставлении, может потребоваться ручная проверка

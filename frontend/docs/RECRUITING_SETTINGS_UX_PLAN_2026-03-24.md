## 1. Кнопка «Обратно» на дочерних страницах раздела рекрутинга

### Цель
На каждой странице, логически входящей в ветку «Рекрутинг» (карточки с хаба `/company-settings/recruiting` и вложенные пути `/company-settings/recruiting/*`), в **едином месте** (верх контентной области, над заголовком/контентом) показывать кнопку **«Обратно»**.

### Поведение навигации
1. Если переход на страницу выполнен **с хаба рекрутинга** через карточку-ссылку, в `location.state` передаётся явный fallback: вернуться на `/company-settings/recruiting`.
2. Если пользователь попал на страницу **из другого места** (сайдбар, прямая ссылка, закладка), `state` может отсутствовать: тогда используется **`navigate(-1)`** (история браузера), чтобы вернуться туда, откуда пришли, **если** это применимо.
3. Если явного `state` нет и откат по истории нежелателен/невозможен, в коде документируется fallback на хаб рекрутинга (минимальная эвристика: при отсутствии `recruitingFrom` после `-1` пользователь остаётся на усмотрение браузера; при необходимости доработать отдельным тикетом).

### Страницы (область применения)
- `/company-settings/recruiting/rules`
- `/company-settings/recruiting/stages`
- `/company-settings/recruiting/commands`
- `/company-settings/recruiting/candidate-fields`
- `/company-settings/recruiting/offer-template`
- `/company-settings/scorecard`
- `/company-settings/rating-scales`
- `/company-settings/sla`
- `/company-settings/vacancy-prompt`
- `/interviewers`
- `/candidate-responses`

### Техническая реализация (зафиксировано)
- Компонент `RecruitingSubpageBackBar` + константа пути хаба и тип `RecruitingNavigationState`.
- Расширение `Link` в `router-adapter`: опциональный проп `linkState` → `to: { pathname, state }`.
- Хаб `RecruitingSettingsHub`: все карточки передают `linkState={{ recruitingFrom: '/company-settings/recruiting' }}`.

## 2. Страница правил привлечения (эталон 3001: `/company-settings/recruiting/rules`) — **полностью к переработке**

### Статус в новом плане
Текущий экран на 3001 и перенесённый функционал в миграции рассматриваются как **временный/наследуемый UI**. В целевом продукте страница **подлежит полной переработке** (информационная архитектура, модель данных, связи с остальной системой).

### Глобальные изменения страницы (смена логики, не только UI)
В плане зафиксировано: **логика и назначение экрана в новой версии предполагаются иными**, чем у текущего `/company-settings/recruiting/rules` на 3001. Перенос «как есть» в миграции — только временная мера; **целевая реализация не является эволюцией того же сценария**, а подразумевает **другую предметную модель и поведение**.

Что это означает для глобального ТЗ:

| Область | Что меняется концептуально |
|--------|---------------------------|
| **Роль страницы** | Не обязательно «список правил с бонусами в таблице»: может быть хаб, мастер настройки, визард или связка нескольких сущностей; границы ответственности экрана пересматриваются. |
| **Модель данных** | Сущности «правило привлечения», поля, привязки к источникам и бонусам могут быть **заменены или разложены** на другие типы записей (например, политики, профили каналов, матрицы условий). Наследование старых записей — отдельная задача (миграция/маппинг). |
| **Потоки пользователя** | Сценарии создания, редактирования, приоритизации и аудита **не копируют** текущий поток 3001, если он не согласован с новой логикой; допускается пошаговая настройка, черновики, согласование. |
| **Интеграции и побочные эффекты** | Связь с расчётом бонусов, заявками, воронкой, биллингом и отчётами может быть **переопределена**: не «та же кнопка сохранить», а явные контракты API и события. |
| **Права и видимость** | Кто видит и редактирует какие блоки — может отличаться от плоского списка на 3001. |

**Итог для команды:** при оценке работ по новой странице **не опираться** на объём «рефакторинга текущего UI»; закладывать **отдельный проект/эпик** с нуля по продуктовой постановке, с явным разделом «отличия от логики 3001».

### Важно заложить в ТЗ переработки (содержательное разбиение)
При проектировании новой версии явно разделить:

| Аспект | Что описать |
|--------|-------------|
| **Аспекты и поля привлечения** | Какие измерения источника/канала/кампании/роли участника фиксируются; обязательные и опциональные поля; справочники vs свободный ввод. |
| **Механизмы** | Как срабатывают правила: триггеры, порядок применения, совместимость с бонусами/штрафами, валюты, периоды действия. |
| **Процессы** | Как правила встраиваются в процесс найма (этапы, заявки, оффер, закрытие вакансии); кто инициатор изменений; аудит. |
| **Связь с фильтрацией и отсевом** | Явные ссылки на правила/настройки, описанные **в других разделах** системы (статусы воронки, причины отказа, SLA, источники и т.д.): не дублировать бизнес-логику, а **связывать** и отображать зависимости. |
| **Прочие параметры** | Приоритеты, активность, условия в свободной форме, локализация, права доступа, миграция со старых записей. |

### UI на миграции (краткое обозначение)
На текущей странице правил в `fe_migration` добавляется **видимый информационный блок** (баннер/каллаут), что раздел **запланирован к переработке**, со ссылкой на этот документ (относительный путь в репозитории).

## 3. Команды workflow: «единые» vs «индивидуальные»

### Цель
На `/company-settings/recruiting/commands` добавить переключатель (в виде **галочки** / логического флага) режима:

- **Включено (единые):** один согласованный набор команд применяется **ко всем рекрутерам** одинаково (на уровне компании / единого профиля workflow).
- **Выключено (индивидуальные):** предполагается возможность **разных** наборов или привязок **по рекрутерам** (и при необходимости по процессам и т.д. — уточняется при подключении API).

### Реализация на текущем этапе
- Состояние в компоненте настроек команд (мок), подпись и поясняющий текст рядом с чекбоксом.
- Сохранение только в клиентском состоянии (до появления API).

### Будущее развитие (отдельный эпик после текущей миграции)

Целевая эволюция страницы `/company-settings/recruiting/commands` (и смежных экранов workflow):

| Направление | Описание |
|-------------|----------|
| **Конструктор процессов (n8n-подобно)** | Возможность собирать **workflow-процессы** из узлов (триггеры, действия, условия, интеграции), а не только список текстовых команд в чате. Визуальный редактор графа, версионирование, тестовый прогон — уточняется в отдельном ТЗ (объём сопоставим с отдельным продуктовым модулем). |
| **Отказ от «системных команд» в UI** | Блоки с фиксированными `/add`, `/del` и т.п. **убрать из настроек** как отдельную сущность: поведение ядра чата/файлов описывать в документации продукта или вынести в системные политики, не смешивая с пользовательскими workflow. |
| **Индивидуальные воркфлоу по доменам HR** | Настройки разбить по **блокам / контекстам** (каждый — свой набор или граф процессов): **Recruiting (рекрутинг)**, **HR Ops**, **L&D**, **C&B (Compensation & Benefits)**, **HR PR / внутренние коммуникации**, **Onboarding** и др. У домена — свои триггеры, роли, видимость в интерфейсе; при необходимости общие переиспользуемые под-процессы (библиотека узлов). |
| **Права и аудит** | Кто может менять workflow какого блока; журнал публикаций и откатов. |

**Интеграция с п. «единые / индивидуальные»:** чекбокс на текущей странице — временный мок; в целевой модели переключатель «единый профиль / по рекрутеру» дополняется выбором **домена (блока)** и привязкой процесса к контексту пользователя или роли.

**UI на миграции:** на странице команд добавлен **каллаут** с отсылкой к этому разделу; после появления ТЗ по n8n-подобному конструктору — заменить/расширить макет страницы под вкладки по блокам и редактор графа.

## 4. Дополнительные поля кандидатов: единый профиль и индивидуальные настройки по вакансии

### Два уровня (целевая модель в новой версии)
Эталон списка полей на 3001: `/company-settings/candidate-fields`. В **новой** версии поведение должно покрывать **оба** сценария:

| Уровень | Где настраивается | Смысл |
|--------|-------------------|--------|
| **Единая «история» / общий профиль** | `/company-settings/recruiting/candidate-fields` (миграция: `fe_migration`) | Состав полей, порядок, типы, обязательность и **базовая** видимость для **всех вакансий** компании — единый справочник и правила по умолчанию. |
| **Индивидуальные случаи** | Карточка вакансии, вкладка дополнительных полей | Переопределения под конкретную вакансию: какие из общих полей включены/скрыты, порядок при необходимости, локальные ограничения — **в контексте одной вакансии**. |

**Пример глубокой ссылки на индивидуальный контекст (миграция, 3002):**  
`/vacancies?vacancy=4090046&mode=view&tab=additionalFields`  
(параметры: `vacancy` — id вакансии, `tab=additionalFields` — вкладка дополнительных полей.)

### Связь уровней
- Настройки компании задают **канонический набор** полей и дефолтную видимость.
- Страница вакансии может **наследовать** этот набор и **тонко настраивать** под вакансию без дублирования определений полей (точная модель наследования/override — на стороне API и ТЗ бэкенда).
- История изменений (аудит) может вестись **отдельно** на уровне компании и на уровне вакансии — уточняется при проектировании API.

### UI на странице настроек компании (`/company-settings/recruiting/candidate-fields`)
- Информационный блок (каллаут/текст), что здесь — **единый профиль для всех вакансий**, а индивидуальные сценарии — на карточке вакансии (с указанием примера URL).
- В колонке **«Действия»** — кнопка-**иконка глаза**, открывающая **модальное окно**:
  - Список **всех вакансий** (мок), где для каждой указано: поле **доступно / недоступно** (переключатель).
  - Заголовок модалки: название поля + пояснение про видимость **по вакансиям** в рамках общего профиля (до API).

### Данные (текущий этап)

- Мок **полей кандидатов** в `CandidateFieldsSettings` расширен: не только название/тип/обязательность, но и `fieldKey`, описание, `multiselect` / `checkbox`, активность, значение по умолчанию (см. таблицу «Полный перечень» ниже).
- Мок-список вакансий (id, название).
- Локальный стейт видимости `fieldId → { vacancyId → boolean }` (или эквивалент).
- Экран вакансии `tab=additionalFields` — отдельная задача: синхронизация с полями из настроек компании (после API).

### Полный перечень свойств поля и типов (чтобы не упустить в ТЗ и API)

Ранее в моке фигурировали только название, тип из ограниченного списка, обязательность и placeholder — этого **недостаточно** для продукта и интеграций. Ниже — ориентир для бэкенда и UI (часть может быть «фаза 2»).

**Типы значения (расширяемый enum):**

| Тип | Назначение |
|-----|------------|
| `text` | Однострочный текст |
| `textarea` | Многострочный текст |
| `number` | Число (целое или дробное — уточнить) |
| `date` | Дата (и при необходимости диапазон дат) |
| `url` | Ссылка с валидацией URL |
| `select` | Один вариант из справочника (опции задаются в настройках поля) |
| `multiselect` | Несколько вариантов из справочника |
| `checkbox` | Да/нет (булево) |
| `file` | Вложение (если нужно в процессе) — опционально, отдельный тикет |

**Метаданные и поведение поля (хранение на уровне компании):**

| Свойство | Описание |
|----------|----------|
| `id` | Стабильный UUID/число сущности в БД |
| `fieldKey` | Уникальный **латинский** ключ для API, отчётов, интеграций (snake_case), не менять без миграции |
| `name` | Отображаемое название (локализуемое при необходимости) |
| `description` | Подсказка / описание для рекрутера и кандидата |
| `type` | См. таблицу типов |
| `required` | Обязательность при заполнении |
| `placeholder` | Подсказка в инпуте (где применимо) |
| `defaultValue` | Значение по умолчанию (строковое представление; для checkbox — true/false) |
| `options` | Список вариантов для `select` / `multiselect` (и при необходимости связь со справочником) |
| `validation` | Правила: min/max, маска, regex, max length — **отдельный объект** по мере необходимости |
| `order` | Порядок в форме |
| `isActive` | Мягкое отключение поля без удаления истории |
| `visibility` | Базовая видимость по вакансиям + наследование на карточке вакансии (см. выше) |
| `audit` | Кто и когда менял определение поля (и отдельно — значения у кандидата) |

**Уровень вакансии (`/vacancies?...&tab=additionalFields`):** переопределения видимости, обязательности, порядка относительно глобального профиля, при необходимости — локальные подсказки (без дублирования `fieldKey`).

## 7. Критерии готовности

- С хаба рекрутинга переход на любую дочернюю страницу и **«Обратно»** возвращает на хаб.
- С других входов «Обратно» использует историю, не ломая навигацию.
- Чекбокс на странице команд отображается и меняет локальное состояние; каллаут отсылает к §3 (будущие workflow по доменам HR и конструктор процессов).
- У каждого поля кандидатов есть действие «глаз» и модалка со списком вакансий; на странице настроек понятно разделение **единого профиля** и **индивидуальных** настроек на вакансии (`/vacancies?...&tab=additionalFields`).
- У каждой строки истории промпта есть открытие модалки с полным текстом.
- Документ в репозитории отражает переработку rules, **глобальную смену логики** (не только макет) и структуру будущего ТЗ.

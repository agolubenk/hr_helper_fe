## Фаза 1 — Инфраструктурный аудит (делать ПЕРВЫМ)

**Цель:** убедиться, что сборка и env-переменные работают корректно до начала переноса логики.

### 1.1 Аудит ENV-переменных
🟢 **ЗАКРЫТО ПО СВОДКЕ** — см. **`fe_migration/docs/DETAILED_PLAN_STATUS.md`** (фаза 1.1): в `fe_migration/src` публичные ключи через `VITE_*` / `import.meta.env`; поддерживать при новых переменных.

В `frontend old` используются `NEXT_PUBLIC_*`. В `fe_migration` должны быть `VITE_*`.

**Что делать:**
1. Запустить `grep -r "NEXT_PUBLIC_" fe_migration/src/` — найти все вхождения
2. Заменить все `process.env.NEXT_PUBLIC_*` на `import.meta.env.VITE_*`
3. Обновить `.env`, `.env.example` — переименовать ключи
4. Проверить `fe_migration/src/lib/api.ts` — там основные API-вызовы (в проекте нет корневого `src/api.ts`)

**Ориентир:** после правки `npm run build` в `fe_migration/` должен завершаться без ошибок.

### 1.2 Page Titles (document.title)
🟢 **ЗАКРЫТО ПО СВОДКЕ** — см. **`DETAILED_PLAN_STATUS.md`** (фаза 1.2): `AppLayout` задаёт заголовок из `pageTitle`; без оболочки — `useDocumentTitle` на страницах входа/сброса пароля. Добавлять для **новых** маршрутов по тому же паттерну.

В Next.js использовался `metadata` / `<title>` на уровне layout. В Vite — нужен `document.title` в каждой странице.

**Что делать:**
```tsx
// Пример паттерна — добавить в каждый *Page.tsx
useEffect(() => { document.title = 'Вакансии — HR Helper'; }, []);
```

**Приоритет страниц для добавления title:**
1. `/workflow` → "Workflow — HR Helper"
2. `/vacancies` → "Вакансии — HR Helper"
3. `/company-settings` → "Настройки — HR Helper"
4. `/ats` → "ATS — HR Helper"
5. Остальные по аналогии

### 1.3 Проверка router-adapter.tsx
🟡 **СКОРРЕКТИРОВАТЬ ПРИ НЕОБХОДИМОСТИ**

Если в `frontend old` есть страницы с `useSearchParams`, `useRouter`, `usePathname` — убедиться, что в `fe_migration/src/router-adapter.tsx` есть соответствующие экспорты.

```bash
grep -r "useSearchParams\|useRouter\|usePathname" "frontend old/" --include="*.tsx"
```

## Фаза 13 — API вместо моков + единый слой моков

🔴 **РЕАЛИЗОВАТЬ — задача §8** (после контракта с бэкендом)

### 13.1 Единый подход к мокам (до и во время перехода на API)

**Цель:** не размазывать фиктивные данные по компонентам; все моки — через **явные модули** и при необходимости **тонкие обёртки** (`getXxx()`), чтобы в одном месте переключить `mock | api`.

| Принцип | Описание |
|---------|----------|
| **Источник данных** | Моки живут в `*Mocks.ts`, `*mocks.ts`, `mocks/` фич или `src/mocks/` — см. **`fe_migration/docs/MOCK_SERVICES.md`** (реестр и правила). |
| **Не смешивать** | Компоненты страниц не содержат больших массивов данных — только импорт из моков/сервиса. |
| **Fallback** | `quickButtonsStorage` и аналоги localStorage — **fallback** при ошибке/отсутствии API, как в таблице ниже. |
| **После API** | Те же обёртки вызывают `api.ts` / React Query; моки остаются для Storybook, e2e и офлайн-режима по решению команды. |

### 13.2 Замена моков профиля и рядом на API

| Компонент | Mock → API |
|-----------|-----------|
| ProfileEditForm | `userMocks` → `GET/PUT` профиля (контракт) |
| QuickButtonsPage | `quickButtonsStorage` → `GET/PUT` быстрых кнопок; **fallback** — `quickButtonsStorage` |
| SocialLinksManager | локальный state / моки → `GET/PUT` соцсетей |
| IntegrationsPage | `integrationsMocks` → `GET/PUT` интеграций |

**Прочие зоны на моках до бэкенда:** бенчмарки (`benchmarksMocks.ts` и др.), ATS (`features/ats/mocks.ts`), часть company-settings — перевод на API по контрактам **фаза 13**, без удаления моков в один день (параллельно или feature-flag).

Подробнее: **`fe_migration/docs/MOCK_SERVICES.md`**.
